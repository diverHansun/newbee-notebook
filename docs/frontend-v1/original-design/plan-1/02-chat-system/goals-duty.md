# 聊天系统 -- 目标与职责

## 1. 设计目标

### 1.1 核心目标

为用户提供基于 Notebook 的多轮对话能力，支持四种交互模式（chat、ask、explain、conclude），通过 SSE 流式传输实现实时的 AI 响应展示，并管理多个独立的聊天会话。

### 1.2 具体目标

- 流式响应：通过 SSE 接收后端的增量内容，实时渲染到聊天界面，减少用户等待感知
- 多模式支持：chat（通用对话）和 ask（文档问答）在主聊天面板展示；explain（解释选中文本）和 conclude（总结选中文本）在独立的浮动卡片中展示
- 多会话管理：同一 Notebook 内支持多个独立的聊天会话，可创建、切换和删除
- 来源引用：ask 模式的 AI 回复可包含文档来源引用，用户可识别回答所基于的文档片段
- 可取消：进行中的流式响应可被用户主动取消

---

## 2. 职责

### 2.1 SSE 流式通信

建立与后端 SSE 端点的连接，解析服务端推送的事件流，处理以下事件类型：

- `start`：流开始，携带 `message_id`，用于关联后续事件和取消操作
- `content`：增量内容片段（`delta`），追加到当前消息文本中
- `sources`：RAG 来源引用数组，在流结束前推送
- `done`：流正常结束
- `error`：流异常，携带 `error_code` 和 `message`
- `heartbeat`：心跳保活，每 15 秒一次，不影响 UI

维护连接生命周期：建立连接、处理事件、正常关闭、异常断开重试（或提示用户）。

### 2.2 消息管理

管理当前会话的消息列表：

- 用户发送消息时立即追加到列表（乐观更新）
- 流式响应期间创建一条 assistant 消息并持续更新其内容
- 流结束后标记消息为完成状态，附加来源引用
- 支持按模式过滤消息（chat/ask 与 explain/conclude 分别展示）
- 消息历史从后端加载，支持分页

### 2.3 多会话管理

- 进入 Notebook 时加载最新会话（调用 `sessions/latest` 端点）
- 无会话时，首次发送消息自动创建新会话
- 提供会话列表供用户切换
- 支持删除会话（需二次确认）
- 会话切换时清空当前消息列表并加载目标会话的消息历史

### 2.4 模式分发与 UI 映射

根据消息的模式类型决定展示位置：

- **chat/ask 模式**：消息显示在中间面板的主聊天区域
- **explain/conclude 模式**：消息显示在独立的可拖动、可调整大小的浮动卡片中

模式由消息发送时的上下文决定：
- 用户在聊天输入框主动输入 → chat 或 ask 模式（由用户切换或系统自动判断）
- 用户在 Markdown 查看器中选中文本并点击操作菜单 → explain 或 conclude 模式，携带 `context`（`document_id` + `selected_text`）

### 2.5 流式取消

在流式响应进行中时，允许用户取消：
- 向后端发送取消请求（`POST /chat/stream/{message_id}/cancel`）
- 立即停止客户端的流式内容追加
- 已接收的部分内容保留在消息中

## 3. 非职责

以下功能明确不属于本模块的职责范围：

- **文本选择检测**：检测用户在 Markdown 查看器中选中文本的行为属于文本选择交互模块。本模块只接收该模块传递的 `context`（document_id + selected_text）并构建请求
- **Markdown 渲染**：AI 回复内容中的 Markdown 渲染由 Markdown 查看器模块完成。本模块将 AI 回复文本传递给 MarkdownViewer 组件，不自行处理 Markdown 语法
- **文档内容获取与展示**：来源引用中的文档跳转逻辑（如点击引用跳转到文档对应位置）不属于本模块
- **用户认证**：请求中的认证信息（如有）由 API 层统一处理
- **聊天记录持久化**：消息持久化由后端完成，本模块只负责从后端加载和向后端发送
- **浮动卡片的拖拽与缩放物理行为**：explain/conclude 卡片的拖拽、缩放、折叠等交互行为的具体实现，可作为通用 UI 组件（如 DraggableCard）独立处理，不与聊天逻辑耦合
