# 书签 422 优化方案

> 本文档基于 [problem-analysis.md](problem-analysis.md) 的结论，给出可执行的优化路线。

---

## 1. 优化目标

1. 消除用户在正常操作下遭遇不可解释 422 的体验问题。
2. 保持后端请求契约清晰，避免无限制放宽导致数据质量下降。
3. 提升错误可观测性，让前端能做明确提示而不是通用失败文案。
4. 补齐关键测试，防止回归。

---

## 2. 约束与原则

1. 现有后端契约约束：`anchor_text` 最大 500 字符。
2. 书签语义优先：`anchor_text` 代表锚点文本，不建议无限增长。
3. 用户体验优先：失败要可预期、可解释、可引导。
4. 改造渐进：先修交互失败，再考虑数据模型增强。

---

## 3. 方案选型

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| A. 仅后端放宽长度上限 | 将 `anchor_text` 上限从 500 提高到更大值 | 改动少，短期可降低 422 | 仍缺少前端边界提示；锚点文本可能过长，影响后续展示与匹配 |
| B. 仅前端限制选择长度 | 提交前校验长度，超长直接提示用户缩短选区 | 立即消除该交互路径 422 | API 调用方若不是前端，仍可能触发 422；错误语义仍不统一 |
| C. 组合方案（推荐） | 前端长度约束 + 后端错误语义标准化 + 测试补齐 | 体验与契约兼顾，治理完整 | 实施工作量高于单点修复 |

推荐采用方案 C。

---

## 4. 推荐改造方案（分阶段）

### 阶段 1：前端边界治理（快速止损）

目标：避免可预期失败请求直接打到后端。

建议动作：

1. 在阅读器书签提交前校验 `selectedText.length`。
2. 超过 500 时不发请求，直接提示“选中文本过长，请缩短后再添加书签”。
3. 失败提示分级：
   - 输入超长：明确提示长度问题
   - 其他异常：保留通用失败提示

交付结果：

- 用户不再在该场景看到“莫名失败”。
- 服务端该类 422 显著减少。

### 阶段 2：后端错误语义统一（接口治理）

目标：让前端与其他调用方能稳定识别错误原因。

建议动作：

1. 对书签创建接口的校验错误建立统一错误码映射（例如 `E_MARK_ANCHOR_TOO_LONG`）。
2. 明确区分两类 422：
   - 请求体校验失败（字段长度/类型）
   - 业务状态不允许（文档未就绪）
3. 在 API 文档中补充错误码说明与触发条件。

交付结果：

- 前端可根据错误码精准提示。
- 日志和监控可按错误类型聚合。

### 阶段 3：锚点准确性增强（质量提升）

目标：降低长文本与复杂排版下锚点偏移不准风险。

建议动作：

1. 优化 `char_offset` 计算策略，避免 `indexOf` 未命中时直接回退到 0。
2. 对跨节点选区、空白折叠场景增加偏移校准逻辑。
3. 增加异常回退策略：无法可靠定位时拒绝创建并提示重新选择。

交付结果：

- 书签定位稳定性提升。
- 降低“创建成功但跳转位置不准”的隐性问题。

---

## 5. 测试计划

### 5.1 前端测试

1. 选区长度小于等于 500：提交成功并刷新书签列表。
2. 选区长度大于 500：不发请求并展示明确提示。
3. 后端返回长度相关错误码：前端展示定向提示。

### 5.2 后端测试

1. 合法请求返回 201。
2. `anchor_text` 超长返回校验错误（包含明确错误语义）。
3. 文档未就绪返回业务 422（与字段校验区分）。

### 5.3 联调回归

1. Markdown Viewer 内进行短选区书签创建，确保成功。
2. 大段选区点击书签，确保有清晰提示且无误导性失败。
3. 书签列表、跳转、删除等相关能力无回归。

---

## 6. 验收标准

1. 用户在超长选区场景下不再遭遇“无说明的创建失败”。
2. 服务端日志中该类可预期 422 明显下降。
3. 前后端测试覆盖新增场景并通过。
4. 书签成功路径（创建、展示、跳转）保持稳定。

---

## 7. 风险与回滚

### 7.1 风险

1. 前端长度计算与后端长度计算口径不一致（空白字符处理差异）。
2. 错误码改造若未同步前端，可能出现提示降级。
3. 偏移计算策略调整可能影响既有标记定位。

### 7.2 回滚策略

1. 前端提示逻辑可独立回滚，不影响后端接口。
2. 后端错误码映射可保持向后兼容旧 `detail` 结构。
3. 偏移计算增强采用开关或渐进发布，异常时回退旧逻辑。
