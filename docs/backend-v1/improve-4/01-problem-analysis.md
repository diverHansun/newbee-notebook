# Improve-4 问题分析

## 1. 分析范围

本分析覆盖 improve-3 集成测试中确认的三类问题：

1. RAG 模式在文档未就绪时返回 500。
2. 文档处理过程中状态未体现 `processing`。
3. 脚本目录职责和命令入口不统一。

分析基线来源：

- `docs/backend-v1/improve-3/test-report.md`
- `docs/ai-core-v1/08-error-handling.md`
- `docs/ai-core-v1/04-api-design.md`
- 当前实现代码（`chat.py`、`chat_service.py`、`document_tasks.py`、`document_repo_impl.py`）

## 2. 问题一：文档未就绪时返回 500

### 2.1 现象

在文档仍处于 `uploaded/pending/processing` 期间，调用 `ask`（以及相关依赖检索路径）时出现 `500 Internal Server Error`，响应体不稳定，缺少可消费字段。

### 2.2 根因

1. 路由层错误映射不完整，仅处理部分异常类型，未覆盖实际运行期异常。
2. 模式前置校验未统一覆盖所有 RAG 场景，导致异常在底层检索/索引调用处抛出。
3. 未接入 ai-core-v1 约定的统一业务错误格式（`error_code/message/details`）。

### 2.3 影响

1. 前端无法稳定识别“处理中”语义，无法给出明确提示。
2. 用户侧表现为系统错误，误导为服务故障而非文档未就绪。
3. 与既有错误码规范脱节（`E4001` 未落地）。

## 3. 问题二：处理中状态不可见

### 3.1 现象

文档轮询表现常为 `uploaded -> completed` 直接跳变，无法观察到 `processing`。

### 3.2 根因

1. 任务内虽已调用状态更新为 `processing`，但状态写入与长耗时处理共用同一事务边界。
2. 会话在任务末尾才提交，导致处理中间态对外不可见。
3. `pending` 状态存在于模型/接口定义中，但主流程设置不足，未体现“排队中”阶段。

### 3.3 影响

1. 前端无法区分“已入队未执行”和“正在执行”。
2. 用户对 10 分钟级处理任务缺少进度反馈，体验不稳定。
3. 与 improve-1 设计中的状态流转约定不一致。

## 4. 问题三：脚本分层与入口不一致

### 4.1 现象

1. 脚本已分布在 `scripts` 与 `medimind_agent/scripts`，但职责边界未制度化。
2. 文档中出现多种命令入口（`python scripts/...` 与 `python -m medimind_agent.scripts...` 并存）。
3. `frontend/scripts` 作为未来层尚未定义约束，后续容易继续混放。

### 4.2 根因

1. 缺少“目录职责 + 调用方式 + 兼容策略”的统一规范文档。
2. 历史阶段文档和脚本演进节奏不一致，命令示例未收敛。

### 4.3 影响

1. 新成员执行脚本成本高，容易误用入口。
2. 自动化脚本、CI、运维命令复用困难。
3. 随前端接入推进，跨端脚本组织风险上升。

## 5. 总结：问题本质

本阶段问题本质不是“核心链路不可用”，而是“系统语义没有稳定对外”：

1. 错误语义未标准化（导致 500）。
2. 状态语义未即时可见（导致不可观测）。
3. 运维语义未目录化（导致入口歧义）。

Improve-4 的工作重点应是“统一语义契约”，为后续前端接入和联调提供稳定边界。
