# 文档处理模块 - 测试说明

## 1. Test Scope (测试范围)

### 1.1 测试覆盖的职责

| 职责 | 测试重点 |
|------|----------|
| 格式识别与路由 | 不同文件类型正确分发到对应转换器 |
| 内容转换 | 转换结果包含预期的 Markdown 结构 |
| 转换结果持久化 | 文件正确写入,路径正确记录 |
| 状态管理 | 状态流转符合预期 |
| 内容查询 | API 正确返回内容 |

### 1.2 不在测试范围

| 内容 | 原因 |
|------|------|
| MinerU 内部转换逻辑 | 外部服务,不属于本模块 |
| MarkItDown 库内部逻辑 | 第三方库,不属于本模块 |
| 文件上传流程 | 由 DocumentService 负责 |
| RAG 索引流程 | 由 RAG 模块负责 |
| 前端渲染效果 | 由前端负责 |

## 2. Critical Scenarios (关键场景)

### 2.1 正常转换场景

#### 场景 2.1.1: PDF 文档转换成功

**前置条件**:
- MinerU API 服务可用
- 有效的 PDF 文件已上传

**预期行为**:
- 调用 MinerU API 进行转换
- 转换结果保存为 Markdown 文件
- Document.status 更新为 COMPLETED
- Document.content_path 指向正确的文件路径
- Markdown 内容包含标题结构

#### 场景 2.1.2: Word 文档转换成功

**前置条件**:
- 有效的 DOCX 文件已上传

**预期行为**:
- 调用 MarkItDown 进行转换
- 转换结果保存为 Markdown 文件
- Document.status 更新为 COMPLETED

#### 场景 2.1.3: 纯文本文件处理

**前置条件**:
- TXT 或 MD 文件已上传

**预期行为**:
- 直接读取文件内容
- 内容保存到目标位置
- Document.status 更新为 COMPLETED

### 2.2 错误处理场景

#### 场景 2.2.1: MinerU 服务不可用

**前置条件**:
- MinerU API 服务未启动或不可达

**预期行为**:
- Document.status 更新为 FAILED
- Document.error_message 包含服务不可用信息
- 不产生部分转换的文件

#### 场景 2.2.2: 原始文件不存在

**前置条件**:
- Document 记录存在
- file_path 指向的文件已被删除

**预期行为**:
- Document.status 更新为 FAILED
- Document.error_message 包含文件不存在信息

#### 场景 2.2.3: 转换超时

**前置条件**:
- PDF 文件过大,转换时间超过阈值

**预期行为**:
- Document.status 更新为 FAILED
- Document.error_message 包含超时信息

#### 场景 2.2.4: 不支持的文件格式

**前置条件**:
- 上传了不支持的文件格式(如 .exe)

**预期行为**:
- Document.status 更新为 FAILED
- Document.error_message 说明格式不支持

### 2.3 内容查询场景

#### 场景 2.3.1: 查询已完成文档的内容

**前置条件**:
- Document.status = COMPLETED
- content_path 对应的文件存在

**预期行为**:
- API 返回 200
- 响应包含完整的 Markdown 内容
- 响应包含正确的元数据(page_count, content_size 等)

#### 场景 2.3.2: 查询处理中文档的内容

**前置条件**:
- Document.status = PROCESSING

**预期行为**:
- API 返回 400
- 响应说明文档尚未准备好

#### 场景 2.3.3: 查询不存在的文档

**前置条件**:
- document_id 在数据库中不存在

**预期行为**:
- API 返回 404
- 响应说明文档不存在

## 3. Integration Points (集成点测试)

### 3.1 与 MinerU API 的集成

**验证重点**:
- HTTP 请求格式正确(multipart/form-data)
- 请求参数完整(backend, lang_list, return_md)
- 响应解析正确(提取 md_content 和 images)
- 超时处理正确(300 秒)

**失败场景处理**:
- 连接超时: 标记任务失败,记录错误
- 响应解析失败: 标记任务失败,记录原始响应
- HTTP 错误状态码: 标记任务失败,记录状态码

### 3.2 与文件系统的集成

**验证重点**:
- 目录创建正确(data/documents/{id}/)
- 文件写入完整(无截断)
- 文件权限正确(可读)
- 路径格式正确(相对路径)

**失败场景处理**:
- 磁盘空间不足: 标记任务失败,清理部分文件
- 权限不足: 标记任务失败,记录权限错误

### 3.3 与 Celery 任务队列的集成

**验证重点**:
- 任务正确入队
- Worker 正确获取并执行任务
- 任务状态可追踪
- 任务失败时状态正确更新

**失败场景处理**:
- Worker 崩溃: 任务保持在队列,等待重新分配
- 任务执行异常: 状态更新为 FAILED

### 3.4 与数据库的集成

**验证重点**:
- Document 状态更新事务完整
- content_path 与文件系统一致
- 并发更新不冲突

**失败场景处理**:
- 数据库连接失败: 任务重试
- 事务冲突: 使用乐观锁重试

## 4. Verification Strategy (验证策略)

### 4.1 单元测试

**测试对象**:
- 转换器选择逻辑
- 文件路径生成逻辑
- 状态流转逻辑

**Mock 策略**:
- Mock MinerU API 调用,返回预设的转换结果
- Mock MarkItDown 库调用,返回预设的转换结果
- Mock 文件系统操作,验证调用参数

**适合自动化**: 是

### 4.2 集成测试

**测试对象**:
- 完整的转换流程(从触发到完成)
- API 端点的请求/响应

**环境要求**:
- 本地文件系统可用
- 测试数据库可用
- MinerU API 可用(或使用 Mock 服务)

**Mock 策略**:
- 使用轻量级的 Mock MinerU 服务
- 使用测试数据库(SQLite 或 PostgreSQL 测试实例)

**适合自动化**: 是

### 4.3 端到端测试

**测试对象**:
- 完整流程: 上传文件 -> 触发转换 -> 查询内容

**环境要求**:
- 完整的 Docker 环境(包括 MinerU)
- 真实的 PostgreSQL 数据库

**验证方式**:
- 上传样本 PDF 文件
- 等待处理完成
- 验证 API 返回的内容包含预期结构

**适合自动化**: 部分(需要真实服务)

### 4.4 人工验证

**测试对象**:
- 转换质量(Markdown 结构是否合理)
- 表格、公式的转换效果
- 中文内容的处理效果

**验证方式**:
- 准备多种类型的样本文档
- 执行转换后人工检查结果
- 记录转换质量问题

**不适合自动化**: 是(需要主观判断)
