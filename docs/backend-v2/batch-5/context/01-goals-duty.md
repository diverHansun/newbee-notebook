# 中层记忆系统 (Context Compaction) -- 设计目标与职责

本文档定义中层记忆系统的存在意义、职责范围与明确排除的职责。

中层记忆系统位于短期记忆（LLM 上下文窗口内的原始消息流）与持久化存储（数据库中的完整消息历史）之间，解决的核心问题是：**当会话消息累积到接近上下文窗口上限时，如何在不丢失关键语义的前提下，将历史消息压缩为精炼摘要，使对话能够持续进行。**

如果没有此模块，长会话将面临两种结局：要么上下文溢出导致最早的消息被丢弃（信息丢失不可控），要么必须人为限制会话长度（体验退化）。

---

## 一、Design Goals（设计目标）

1. **保障长会话的语义连续性**：即使对话持续数十轮，模型仍能准确理解早期讨论的关键决策、结论和用户偏好。

2. **基于真实 token 用量触发压缩**：使用 tiktoken 进行准确的 token 计数，当上下文消耗达到窗口的 95% 时自动触发压缩，避免基于轮数的粗粒度判定。

3. **对 ContextBuilder 和 AgentLoop 透明**：压缩作为请求处理流程中的前置步骤完成，下游组件无需感知压缩的存在，保持各组件的单一职责。

4. **压缩产物可审计**：所有历史 summary 均持久化到数据库，支持回溯查看某一时刻的压缩结果，但同一时刻模型只看到最近一次压缩的 summary。

5. **压缩成本可控**：summary 输出严格控制在 4000~6000 tokens，避免摘要本身膨胀挤占上下文空间。

---

## 二、Duties（职责）

1. **token 用量评估**：基于 tiktoken 准确计算当前会话 boundary 之后所有消息的 token 总量，与压缩阈值比较，判定是否需要触发压缩。

2. **历史消息压缩**：将旧 summary（如存在）与 boundary 后的消息一起输入当前 LLM，生成新的精炼摘要。

3. **压缩边界管理**：维护 Session 上的 `compaction_boundary_id` 指针，确保 boundary 之前的消息不会进入模型可见的消息列表。

4. **summary 持久化**：将压缩产生的 summary 作为 `message_type=SUMMARY` 的 assistant 消息写入数据库，更新 Session 的 boundary 指针。

5. **前置拦截执行**：在 SessionManager 构建上下文之前，作为独立步骤执行压缩检查与处理，压缩完成后再由 SessionManager 继续正常流程。

---

## 三、Non-Duties（非职责）

1. **不负责上下文组装**：消息列表的构建仍由 ContextBuilder 完成，本模块只负责确保 SessionMemory 中的消息已经过 boundary 过滤。

2. **不负责 side-track 压缩**：EXPLAIN / CONCLUDE 模式使用独立的 side-track，已有 12 条消息硬限制，不在本模块的压缩范围内。

3. **不负责工具消息的管理**：工具调用和结果仅存在于 AgentLoop 的单次执行内存中，不持久化，不参与压缩。

4. **不负责 ContextBudget 的动态调整**：预算配置由外部提供（配置层），本模块仅使用预算中的阈值进行判定。

5. **不负责消息的过期清理或删除**：数据库中的历史消息（包括旧 summary）保持完整，本模块不执行物理删除。
