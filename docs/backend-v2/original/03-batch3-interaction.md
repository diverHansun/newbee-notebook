# 第三批：交互增强层

本批次包含 1 个核心模块，分为笔记和书签两个子功能。依赖第一批的分页系统完成 (书签需要页码定位)。

---

## 模块: Studio 笔记与书签系统

### 目标

在 Studio Panel 中实现独立的笔记编辑器和文档书签引用系统，让用户能够在阅读文档的过程中记录想法、标记关键位置，并在笔记中引用这些位置。

### 概念定义

- **Note (笔记)**: 用户在 Studio Panel 中自由编写的文本内容，与笔记本关联。笔记内容为 Markdown 格式，可包含书签引用。
- **Mark (书签)**: 对文档中特定位置的标记。每个书签记录文档 ID、页码和锚定文本，可独立存在，也可被笔记引用。

### 职责

**笔记子功能**:
- 笔记 CRUD 操作，关联到笔记本
- 笔记内容存储和管理
- 笔记中嵌入书签引用的解析和展示

**书签子功能**:
- 在 Markdown 阅读器中选中文本后创建书签
- 书签记录精确位置信息 (document_id / page_number / anchor_text)
- 书签列表查看和管理
- 从书签跳转到文档对应位置

**笔记-书签关联**:
- 在笔记编辑时插入书签引用
- 笔记中的书签引用可点击跳转到源文档位置

### 非职责

- 不实现笔记的协作编辑或多用户共享
- 不实现笔记的全文搜索 (初期仅列表浏览)
- 不实现笔记内容的 AI 分析 (后续可通过 Skill 扩展)

### 数据模型概要

两个核心实体：
- **notes**: 关联 notebook_id，包含标题和 Markdown 内容
- **marks**: 关联 document_id，包含页码、锚定文本和可选短评；可选关联 note_id

### API 设计方向

笔记 CRUD:
- `POST/GET /notebooks/{id}/notes`
- `GET/PUT/DELETE /notes/{note_id}`

书签 CRUD:
- `POST/GET /documents/{id}/marks`
- `GET /notebooks/{id}/marks`
- `DELETE /marks/{mark_id}`

笔记-书签关联:
- `POST/DELETE /notes/{note_id}/marks/{mark_id}`

### 前端交互

**书签创建流程**: 用户在 Markdown Viewer 选中文本 -> 出现"添加书签"选项 -> 记录位置信息 -> 书签出现在 Studio Panel 列表中

**笔记编辑流程**: 用户在 Studio Panel 创建/打开笔记 -> 编写 Markdown 内容 -> 插入书签引用 (选择已有书签) -> 保存

**跳转流程**: 用户点击书签或笔记中的书签引用 -> Markdown Viewer 跳转到对应文档的对应页码和位置

### 前置依赖

- 第一批的分页系统必须完成: 书签的页码定位依赖文档分页功能
- MinIO 迁移完成有利于书签关联的图片访问，但非强依赖
