# Context 模块：设计目标与职责边界

## 1. 设计目标

### 1.1 双轨上下文隔离

维护两条独立的对话轨道——Main 轨道（Agent/Ask）和 Side 轨道（Explain/Conclude）。两条轨道有明确的可见性规则：Side 可以读取 Main 的历史作为背景上下文，Main 不感知 Side 的存在。这反映了产品语义：解释和总结需要理解主对话的上下文，但主对话不需要知道用户做过哪些解释。

### 1.2 分层压缩

对话历史随交互次数增长，不做任何处理会超出模型上下文窗口。Context 模块实施分层压缩策略，将历史消息分为完整层、压缩层、摘要层三个层级，在信息保留和 token 消耗之间取得平衡。压缩在请求结束后异步执行，不增加用户等待时间。

### 1.3 Token 预算显式分配

对消息链的各组成部分（system prompt、历史消息、工具结果、用户输入、LLM 输出预留）设置显式的 token 预算。预算分配是可配置的，而非硬编码的隐式限制。这使得上下文管理行为可预测、可调优。

### 1.4 工具结果压缩

AgentLoop 在一次请求中可能做多轮工具调用，每轮产出的工具结果会累积在消息链中。Context 模块对工具结果实施"最近完整、较早压缩"的策略：最近一轮的工具结果保留完整文本，更早轮次的工具结果压缩为紧凑摘要，控制单次请求内的上下文膨胀。

### 1.5 与 LlamaIndex 消息格式兼容

Context 模块的输出是 `List[ChatMessage]`（LlamaIndex 的消息类型）。这是 Context 模块与 LLM 层之间的契约——AgentLoop 直接将此列表传给 `llm.achat_with_tools()`，无需任何格式转换。Context 模块不依赖 LlamaIndex 的 Memory 子系统（ChatMemoryBuffer 等），只使用其消息数据类型。

## 2. 职责

### 2.1 双轨内存管理

维护 Main 轨道和 Side 轨道的对话消息。提供按模式类型（Agent/Ask/Explain/Conclude）读取和写入历史的接口。执行可见性规则：为 Explain/Conclude 构建消息链时，将 Main 轨道的近期历史作为只读前缀注入。

### 2.2 消息链构建

根据请求的模式类型，组装完整的消息链。消息链的组成部分：

- 对于 Agent/Ask：Main 轨道历史
- 对于 Explain/Conclude：Main 轨道近期历史（只读，截断后）+ Side 轨道历史

构建过程受 token 预算约束，当历史超出预算时触发压缩。

### 2.3 Token 统计

基于 LLM 的 tokenizer 统计消息的 token 数量。为消息链的各组成部分提供 token 计数，供预算分配决策使用。

### 2.4 分层压缩执行

当历史消息超出预算时，按照分层策略进行压缩：

- 完整层（最近 N 轮）：保留原始消息不做任何处理。
- 压缩层（较早的 M 轮）：保留 user 消息原文，assistant 回复截断为首段。
- 摘要层（更早的消息）：由 LLM 生成一段摘要文本，替换所有原始消息。

摘要层的 LLM 调用在请求结束后异步执行。

### 2.5 工具结果压缩

对同一请求内多轮工具调用的结果进行差异化处理。最近一轮的 tool_result 保留完整 content，更早轮次的 tool_result 替换为截断版本。工具结果的 token 预算上限可配置。

### 2.6 从持久化消息恢复

接收从数据库加载的历史消息列表，重建双轨内存状态。这在 Session 恢复时调用——用户重新打开一个已有会话时，session 模块从数据库取出消息，交给 context 模块重建内存。

## 3. 非职责

### 3.1 消息持久化

Context 模块不负责将消息写入数据库。它是内存中的状态管理，持久化由 session 模块协调。

### 3.2 AgentLoop 执行

Context 模块不知道 AgentLoop 的存在。它不触发 LLM 调用（摘要压缩除外）、不执行工具、不管理流式事件。

### 3.3 模式语义

Context 模块不理解 Agent/Ask/Explain/Conclude 的业务含义。它只知道"Main 轨道"和"Side 轨道"的区分。模式到轨道的映射由调用方（session 模块或 engine 模块）提供。

### 3.4 HTTP 协议

Context 模块不处理 HTTP 请求/响应，不了解 SSE 协议。
