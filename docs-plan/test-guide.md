# test-guide.md

本指南用于指导 AI 如何为**单一模块**撰写 `test.md`，并明确测试在文档体系中的**角色与边界**。

`test.md` 的目标不是穷举测试用例，而是说明：
**如何验证该模块在真实协作环境中是可信的。**

测试在这里是一种工程验证手段，而不是质量口号。

---

## 一、test.md 的核心作用

`test.md` 用于验证模块设计在实践中是否成立。

`test.md` 关注的是**行为是否符合预期**，而不是实现是否精巧。

---

## 二、撰写 test.md 的前置条件

在开始撰写 `test.md` 之前，AI 必须确认：

1. 模块的职责边界已在 `goals-duty.md` 中确认
2. 模块的外部接口和数据流已在 `dfd-interface.md` 中描述
3. 已明确哪些行为对系统是关键的、不可破坏的
4. **已判定模块所属的「模块原型」（见第三节）**

如果上述信息不完整，AI 不应直接生成测试说明，而应提示用户补充设计信息。

---

## 三、模块原型与测试重心

不同性质的模块，其测试的关注点有本质区别。在撰写 `test.md` 之前，AI 必须先判定模块属于哪种原型，再据此确定测试重心。

### 原型分类

| 原型 | 特征 | 典型模块 |
|------|------|----------|
| **纯逻辑模块** | 接收输入、计算、返回输出；无副作用，不依赖外部状态 | compressor, token_counter, selector, context_builder |
| **服务编排模块** | 协调多个组件完成业务流程；自身逻辑较轻，核心价值在于编排顺序和条件分支 | session_service, document_service, chat_service |
| **桥接 / 适配模块** | 将内部抽象连接到外部协议或系统边界；核心价值在于接口格式的正确性 | API routers, MCP connectors, storage adapters |
| **基础设施模块** | 提供环境级能力（数据库初始化、容器编排、迁移脚本）；核心价值在于环境能否正常工作 | DB init scripts, docker-compose, migration scripts |
| **外部依赖封装模块** | 封装第三方服务（LLM、embedding、ASR）；核心价值在于隔离外部变化、提供稳定内部接口 | LLM client, embedding provider, ASR transcriber |

### 原型 → 测试重心映射

| 原型 | 测试重心 | 主要测试类型 | test.md 应侧重 |
|------|----------|-------------|----------------|
| 纯逻辑模块 | 输入输出的正确性、边界条件 | unit | Critical Scenarios（场景覆盖） |
| 服务编排模块 | 组件协作流程、状态流转、异常分支 | unit + integration | Critical Scenarios + Integration Points |
| 桥接 / 适配模块 | **接口契约**——请求格式、响应结构、状态码、协议兼容性 | contract | Integration Points + Contract Specification |
| 基础设施模块 | 环境能否正常启动和运行 | smoke | Verification Strategy（执行环境与前置条件） |
| 外部依赖封装模块 | mock 边界的正确性、降级行为、错误映射 | unit + contract | Critical Scenarios + Integration Points |

### 判定规则

AI 应按以下顺序判断模块原型：

1. 该模块是否直接与外部系统/协议交互？→ **桥接 / 适配** 或 **外部依赖封装**
2. 该模块是否提供环境级能力（非业务逻辑）？→ **基础设施**
3. 该模块是否主要协调其他模块工作？→ **服务编排**
4. 以上皆不是 → **纯逻辑**

一个模块可能兼具多种特征（例如一个 service 既有编排逻辑又有纯计算），此时 AI 应在 test.md 的 Test Scope 中说明各部分分别按哪种原型测试。

---

## 四、AI 在撰写前应与用户澄清的问题

在生成 `test.md` 之前，AI 至少应与用户确认：

- 该模块属于哪种**模块原型**？（AI 可先给出判断，请用户确认）
- 该模块最关键的行为是什么？哪些失败是绝对不可接受的？
- 是否存在需要 mock 的外部依赖？mock 的边界在哪里？
- 当前阶段更关注核心路径验证，还是边界情况覆盖？

若信息不足，AI 可提出候选测试场景，并标注为假设。

---

## 五、test.md 的核心原则

### 1. 测试围绕"职责"，而不是"代码结构"
测试应验证模块是否履行其职责，而不是是否覆盖某个类或函数。

### 2. 测试围绕"交互边界"，而不是内部实现
重点放在：
- 模块输入是否被正确处理
- 模块输出是否符合约定

不关注内部实现细节的变化。

### 3. 测试数量服从测试价值
测试的目标是**建立信心**，而不是追求覆盖率数字。

### 4. 测试类型服从模块性质
不要对所有模块套用同一套测试模板。桥接模块的核心价值是契约稳定性，用单元测试验证它的内部逻辑是错位的；纯逻辑模块的核心价值是计算正确性，用集成测试验证它是浪费的。

---

## 六、test.md 的推荐结构

`test.md` 必须包含以下部分。各部分的详略程度应根据模块原型调整——原型映射表中标注为「侧重」的部分应详细展开，其余部分可精简。

---

### 1. Module Test Profile（模块测试档案）

**所有原型必须包含此部分。** 这是 test.md 的「元数据头」，用于快速建立测试定位。

**要求：**
- 声明模块原型
- 声明主要测试类型
- 声明 mock 边界（哪些依赖会被 mock，哪些使用真实实现）

**示例：**

```markdown
## Module Test Profile

- 模块原型：桥接 / 适配模块（HTTP Router）
- 主要测试类型：contract
- Mock 边界：Service 层全部 mock；HTTP 请求/响应使用 TestClient 真实发送
- 测试归属目录：tests/contract/api/
```

---

### 2. Test Scope（测试范围）

用于明确测试关注的内容和刻意不覆盖的内容。

**要求：**
- 指出测试覆盖的模块职责
- 明确不在本模块测试范围内的行为
- **对于混合原型模块**，分别说明各部分的测试归属

**示例（服务编排模块）：**

```markdown
## Test Scope

覆盖：
- DocumentService 的文档创建、删除、状态流转编排逻辑
- 异常分支处理（存储失败、嵌入服务不可用时的降级行为）

不覆盖：
- StorageBackend 的具体存储逻辑（属于 infrastructure 层的单元测试）
- API 层的请求格式验证（属于 router 层的契约测试）
```

---

### 3. Critical Scenarios（关键场景）

用于列出模块中**不可接受失败的场景**。

**要求：**
- 场景应直接来源于模块职责或数据流
- 每个场景都应有明确的预期结果
- **纯逻辑模块**应侧重此部分，列出核心计算路径和边界条件
- **服务编排模块**应包含正常流程和异常分支两类场景

**示例结构：**

```markdown
## Critical Scenarios

### 正常路径
- 输入有效文档时，返回处理成功状态，且存储和嵌入均被触发

### 异常路径
- 存储写入失败时，不触发嵌入，返回明确错误
- 嵌入服务超时时，文档状态标记为"待重试"而非"失败"
```

---

### 4. Contract Specification（契约规约）

**桥接 / 适配模块 和 外部依赖封装模块 必须包含此部分。** 其他原型可省略。

用于定义模块对外暴露的接口契约——这些契约是该模块对其消费者的承诺，不可随内部重构而改变。

**要求：**
- 列出需要验证的接口端点或协议方法
- 明确请求/响应的格式约定
- 明确错误码/状态码的语义

**示例（HTTP Router）：**

```markdown
## Contract Specification

### POST /api/v1/documents
- 请求体：multipart/form-data，必须包含 file 字段
- 成功响应：201，body 包含 { id, status, created_at }
- 文件类型不支持时：400，body 包含 { detail: "Unsupported file type: ..." }
- 未认证：401

### DELETE /api/v1/documents/{id}
- 文档存在时：204，无 body
- 文档不存在时：404
```

**示例（外部依赖封装）：**

```markdown
## Contract Specification

### LLMClient.complete()
- 对内承诺：始终返回 LLMResponse 或抛出 LLMError，不透传第三方异常
- 超时行为：超过配置的 timeout 后抛出 LLMTimeoutError
- 重试策略：429/5xx 自动重试最多 3 次，其他错误立即失败
```

---

### 5. Integration Points（集成点测试）

用于说明模块与外部模块交互时的验证重点。

**要求：**
- 明确涉及哪些外部模块或接口
- 指出交互失败时的预期行为
- **服务编排模块**应侧重此部分

**示例：**

```markdown
## Integration Points

- 与 StorageBackend 交互：写入失败时 service 应回滚状态而非静默忽略
- 与 EmbeddingProvider 交互：provider 不可用时应标记文档为待重试
- 与 SessionManager 交互：会话过期时应返回明确错误而非空结果
```

---

### 6. Verification Strategy（验证策略）

用于说明测试的执行方式和环境要求。

**推荐内容：**
- 测试运行所需的环境前置条件
- 是否使用模拟（mock）外部依赖，mock 的范围和理由
- 是否需要真实环境集成测试
- 哪些行为更适合自动化测试，哪些需要人工验证
- **基础设施模块**应侧重此部分，详细说明环境依赖和启动条件

**示例（基础设施模块）：**

```markdown
## Verification Strategy

- 需要 Docker 环境运行
- 验证 docker-compose up 后所有服务健康检查通过
- 验证数据库迁移脚本幂等执行（连续运行两次不报错）
- 标记为 smoke 测试，仅在部署流水线中执行
```

该部分不要求绑定具体测试框架或工具。

## 七、各原型的 test.md 侧重速查

| 章节 | 纯逻辑 | 服务编排 | 桥接/适配 | 基础设施 | 外部依赖封装 |
|------|--------|---------|----------|---------|-------------|
| Module Test Profile | **必须** | **必须** | **必须** | **必须** | **必须** |
| Test Scope | 标准 | 标准 | 标准 | 精简 | 标准 |
| Critical Scenarios | **详细** | **详细** | 精简 | 省略 | **详细** |
| Contract Specification | 省略 | 省略 | **详细** | 省略 | **详细** |
| Integration Points | 精简 | **详细** | 标准 | 省略 | 标准 |
| Verification Strategy | 精简 | 标准 | 标准 | **详细** | 标准 |

---

## 八、明确禁止的内容

以下内容不应出现在 `test.md` 中：

- 具体测试代码实现
- 与模块职责无关的测试设想
- 仅以"覆盖率"或"测试数量"为导向的描述
- 重复实现文档中已有的数据流或接口说明

---

## 九、撰写时的约束与注意事项

### 1. 避免测试替代设计
如果在编写测试说明时发现职责不清，应回到设计文档澄清，而不是通过测试"补救"。

### 2. 测试是稳定性的保障，而不是灵活性的阻碍
测试说明应关注**稳定对外行为**，而不是限制内部重构。

### 3. 保持测试与设计同步
当模块职责、数据流或接口发生变化时，AI 应提示同步更新 test.md。

### 4. 模块原型可能演化
随着模块职责变化，原型可能改变（例如一个纯逻辑模块增加了外部依赖）。当原型变化时，AI 应提示重新评估 test.md 的结构和侧重点。

---

## 十、完成后的自检清单（AI 必须执行）

在提交 `test.md` 前，AI 必须确认：

- [ ] 是否已判定模块原型并在 Module Test Profile 中声明？
- [ ] 是否根据原型调整了各章节的详略程度？
- [ ] 是否所有关键职责都有对应的验证场景？
- [ ] 桥接/适配模块是否包含了 Contract Specification？
- [ ] 是否明确了模块与外部交互时的失败处理预期？
- [ ] 是否声明了 mock 边界？
- [ ] 是否避免了与具体实现细节的绑定？
- [ ] 测试归属目录是否与项目测试分类规范一致？

如无法确认，应建议进一步讨论设计。

---

## 十一、与其他文档的关系

```
goals-duty.md    → test.md 验证职责是否被履行
dfd-interface.md → test.md 覆盖关键数据流
test.md          → 指导 tests/ 目录下测试代码的编写
                 → 测试代码的组织规范参见 docs-test/（项目级测试方法论）
```

- test.md 是对 goals-duty.md 的行为层验证
- test.md 应覆盖 dfd-interface.md 中描述的关键数据流
- test.md 中的「测试归属目录」和「主要测试类型」应与 `docs-test/` 中的分类规范保持一致
- 当测试需求推动设计调整时，应回溯并更新前置文档
